## 5. Проблемы стабильности сессий

| Событие | Триггер | Влияние | Восстановление |
|---|---|---|---|
| 401 → очистка токена → перезагрузка | Клик по кнопкам, вызывающим API с истёкшим токеном | Разлогинивание во время теста | Повторный вход через `relogin()` |
| Rate-лимит (429) | >3 попыток входа за короткий период | Не удалось протестировать Охранника, Директора, Админа | 35 сек задержка (недостаточно для 6 ролей) |

**Корневая причина:** `src/frontend/src/services/api.ts:104-121` — при 401 (кроме эндпоинта логина) приложение очищает auth-токен и zustand-состояние, затем перезагружается:

```tsx
if (response.status === 401 && endpoint !== '/api/auth/login' && !isHandling401) {
  isHandling401 = true;
  setTimeout(() => { isHandling401 = false; }, 5000);
  localStorage.removeItem('auth_token');
  // Очистка zustand auth state
  ...
  setTimeout(() => window.location.reload(), 100);
}
```

**Примечание:** Обработчик уже очищает только auth-ключи (не весь localStorage). Потери сессий при тестировании вызваны легитимно истёкшими токенами при переключении ролей. Флаг `isHandling401` предотвращает каскадные 401. Поведение приемлемое.
