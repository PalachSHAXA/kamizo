## TL;DR

Найдено **9 P0** (data leak / data corruption риски), **15 P1** (UX / consistency), **10 P2** (code smell / cleanup).

Самое критичное — **3 находки**, которые надо чинить **до следующего деплоя**:

1. **`cloudflare/src/routes/misc/ratings.ts:35`** — `GET /api/ratings` запрашивает `employee_ratings` БЕЗ фильтра `tenant_id`. Любой авторизованный житель Tenant A видит рейтинги Tenant B, если знает `resident_id`.

2. **`cloudflare/schema_no_fk.sql:1398, 1442`** — синтаксическая ошибка SQL: `category_id TEXT NOT NULL ),` — оборванный `)` без `REFERENCES`. Файл не выполнится. Если кто-то задеплоит no-fk вариант — БД не построится.

3. **`cloudflare/src/routes/requests/work-orders.ts:34-44`** — `GET /api/work-orders` JOIN-ит `users` без `tenant_id` условия. Стартовый `whereClause` не содержит `wo.tenant_id = ?`. Cross-tenant утечка work order'ов.

## 1 · Backend — security & tenant isolation

### P0 — утечки между тенантами

- **B1** ✅ `routes/misc/ratings.ts:35` — `SELECT * FROM employee_ratings WHERE resident_id = ?` — нет `AND tenant_id = ?`. Fix: добавить `AND tenant_id = ?`, биндить из `getTenantId(request)`.
- **B2** ✅ `routes/requests/work-orders.ts:34-44` — нет `WHERE wo.tenant_id = ?` ни в whereClause, ни в JOIN. Fix: инициализировать whereClause `'WHERE wo.tenant_id = ?'`, JOIN: `LEFT JOIN users u ON wo.assigned_to = u.id AND u.tenant_id = wo.tenant_id`. Применить ко всем 3 list/get-by-id запросам.
- **B3** `routes/rentals/guests.ts:38` — условный JOIN `${tenantId ? 'AND u.tenant_id = ?' : ''}`. Fix: tenantId обязателен после auth, всегда фильтр.
- **B4** `routes/meetings/voting.ts:111-112` — подзапрос статистики голосования по `meeting_id` без tenant. Fix: добавить tenant scope.
- **B5** ✅ `routes/misc/announcements-mutations.ts:65-73` — `target_building_id` без проверки tenant. Fix: validate `SELECT id FROM buildings WHERE id = ? AND tenant_id = ?`, 403 если null.
