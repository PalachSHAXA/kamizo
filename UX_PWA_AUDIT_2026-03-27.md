### 🔴 Критические проблемы PWA

**[PWA-1] `user-scalable=no` — доступность сломана**
`maximum-scale=1.0, user-scalable=no` блокирует зум. Нарушает WCAG 1.4.4 и поведение iOS 10+. Люди с плохим зрением не могут увеличить текст.
**Файл**: `src/frontend/index.html`
**Рекомендация**: убрать `user-scalable=no` или поставить `maximum-scale=5.0`.

**[PWA-2] Нет стратегии обновления JWT-токена**
В `authStore.ts` нет visible refresh token flow. Если сессия истекает — пользователь выкидывается на логин, теряя несохранённые данные.
**Рекомендация**: добавить silent token refresh за 5 минут до истечения токена.

**[PWA-3] Orientation lock только `portrait-primary`**
Управляющий за iPad хочет ландшафт. Жёсткая фиксация неудобна.
**Файл**: `src/frontend/public/manifest.json`
**Рекомендация**: убрать `orientation` из манифеста или поставить `any`.

**[PWA-4] Нет принудительной инвалидации кэша при критических миграциях**
При деплое смены схемы данные могут быть stale. Нет механизма принудительного обновления клиентов.
**Рекомендация**: добавить `skipWaiting()` + `clients.claim()` агрессивно, версионировать данные в IndexedDB.

## 2. МОБИЛЬНАЯ ВЕРСТКА — СИСТЕМНЫЕ ПРОБЛЕМЫ

### [MOB-1] 🔴 BottomBar конфликтует с контентом

BottomBar фиксированный, контент имеет `padding-bottom: 90px`. Проблемы:
- Модальные окна с формами (NewRequestModal, ApproveModal) не учитывают BottomBar — поля ввода перекрываются на iOS при открытии клавиатуры
- `100vh` на iOS Safari ≠ видимая область — layout прыгает когда адрес-бар прячется
- На Android Chrome с жестовой навигацией BottomBar "утопает" без `env(safe-area-inset-bottom)`

**Рекомендация**:
```css
.bottom-bar {
  padding-bottom: max(12px, env(safe-area-inset-bottom));
  height: calc(60px + env(safe-area-inset-bottom));
}
```

### [MOB-2] 🔴 Свайп-сайдбар конфликтует с системными жестами iOS

Порог открытия: свайп из зоны `0–25px` от края. iOS 13+ системный жест "назад" живёт в той же зоне. Пользователи случайно открывают сайдбар вместо навигации браузера.
**Файл**: `src/frontend/src/components/layout/Layout.tsx`
**Рекомендация**: зону открытия сместить до `0–15px` или добавить `touch-action: pan-y` на контент.

### [MOB-3] 🟡 Таблицы не адаптированы под мобильный

Таблицы в `/requests`, `/residents`, `/executors` — горизонтальный скролл на мобильном вместо card-вида.
**Рекомендация**: для < 768px рендерить card-layout вместо таблицы.

## 3. РОЛЬ: ЖИТЕЛЬ (RESIDENT)

### Хронология пользователя

```
Установка PWA → Логин → Онбординг (4 шага) → Главная (2 вкладки) → ...
  Смена пароля → Телефон → Контракт → Авто
```

### ✅ Хорошо

- Онбординг через MobileHeader (4 шага) — логичная последовательность
- Dashboard с 2 вкладками (Главная / Заявки) — правильное разделение
- Polling каждые 15 сек для reschedule — реактивный UX без WebSocket

### [RES-1] 🔴 Кнопка "Новая заявка" недоступна за BottomBar

На мобильном FAB (или кнопка внизу страницы) скрыта под BottomBar при длинном списке заявок. Главный CTA жителя — создание заявки — недоступен без скролла вверх.
**Файл**: `src/frontend/src/pages/ResidentDashboard.tsx`
**Рекомендация**: кнопка должна быть `position: fixed` с `bottom: calc(BottomBar-height + 16px)`.

### [RES-2] 🔴 Одобрение заявки = принудительный рейтинг

`ApproveModal` требует поставить оценку при одобрении. Житель хочет просто нажать "ОК, выполнено", но форма рейтинга блокирует действие. Конверсия одобрений снижается — пользователи закрывают модалку.
**Рекомендация**: разделить потоки: "Принять работу" (1 клик) → отдельный баннер "Оцените исполнителя" через 5 мин.

### [RES-3] 🟡 Модальные окна — не bottom sheets на мобильном

`NewRequestModal`, `RequestDetailsModal`, `RescheduleModal` — вероятно центрированные overlays. Центрированный диалог + клавиатура на iOS = критическая проблема (поля уходят за экран).
**Рекомендация**: все модалки на мобильном = bottom sheet. Длинные формы (с фото) = full-page с кнопкой "Назад".

### [RES-4] 🟡 Объявления дублируются на Dashboard и в `/announcements`

3 объявления на главной + отдельная страница. При нажатии на объявление с главной — куда ведёт ссылка?
**Рекомендация**: клик на объявление → детальная страница. Ссылка "Все объявления" → `/announcements`.

### [RES-5] 🟡 "Полезные контакты" — отдельная страница в навигации

Список телефонов — это не регулярное действие. Отдельный пункт меню перегружает сайдбар.
**Рекомендация**: переместить в Profile → "Контакты УК" или в FAQ-блок на Dashboard.

### [RES-6] 🟡 "Договор" в сайдбаре после подписания не нужен

После подписания контракт нужен редко. Постоянный пункт меню создаёт шум.
**Рекомендация**: после `contractSignedAt` — убрать из сайдбара, переместить в Profile → "Документы".

### [RES-7] 🟡 "Оценить сотрудников" в навигации — лишний пункт

Это контекстное действие после завершения заявки, а не регулярная функция.
**Рекомендация**: убрать из навигации, оставить только в потоке после завершения заявки.

### [RES-8] 🔵 Маркетплейс без контекста

"Домашний Маркет" в меню — что это? Нет поясняющей иконки/подзаголовка о том, что это магазин услуг/товаров.

## 4. РОЛЬ: ИСПОЛНИТЕЛЬ (EXECUTOR)

### Хронология пользователя

```
Логин → ExecutorDashboard (список заявок) → Принять → В работе → Завершить
                    ↓
         Schedule (календарь) | Stats | Announcements | Chat
```

### [EXE-1] 🔴 Поиск авто у исполнителя — лишний пункт

В меню исполнителя есть `/vehicle-search`. Задача исполнителя — выполнять ремонтные заявки. Поиск по базе авто — функция охранника/менеджера.
**Рекомендация**: убрать `vehicle-search` из меню executor, оставить только для `security`, `manager`, `admin`.

### [EXE-2] 🟡 Dashboard vs Schedule — два источника правды

`ExecutorDashboard` (список заявок) и `ExecutorSchedulePage` (календарь) — дублируют информацию в разных форматах.
**Рекомендация**: Dashboard = активные заявки сегодня. Schedule = планирование на неделю. Переключение вид "список/календарь" инлайн на одной странице, а не два пункта меню.

### [EXE-3] 🟡 Статистика — отдельная страница избыточна

Ключевые метрики (выполнено сегодня/неделю, рейтинг) должны быть видны на Dashboard.
**Рекомендация**: вынести 3-4 KPI на Dashboard плашками. Страница Stats — только для детального анализа.

### [EXE-4] 🟡 BottomBar исполнителя перегружен

Слишком много вкладок для роли с простым сценарием использования.
**Рекомендация**: BottomBar исполнителя: `Заявки | Расписание | Чат | Профиль`. Stats убрать в профиль.

## 5. РОЛЬ: ОХРАННИК (SECURITY)

### Хронология пользователя

```
Логин → redirect /qr-scanner → [камера] → результат проверки
              ↓
         Vehicle Search | Announcements | Chat
```

### [SEC-1] 🔴 UX для QR-сканера не адаптирован

При логине охранника — redirect на `/qr-scanner`. Правильная логика, но:
- Нет инструкции при первом входе для запроса разрешения на камеру
- Нет режима ручного ввода QR-кода (на случай плохого освещения или поломки камеры)
- Нет истории сканирований текущей сессии

**Рекомендация**:
1. При первом входе — объяснительный экран "Разрешите доступ к камере"
2. Кнопка "Ввести вручную" всегда видна под видоискателем
3. "Последние проходы" — список последних 10 записей в боковой панели

### [SEC-2] 🟡 Vehicle Search для охранника — нужен упрощённый интерфейс

Интерфейс поиска общий с менеджером — он избыточен. Охраннику нужно: ввести номер → Есть/Нет в базе + фото/ФИО хозяина.
**Рекомендация**: отдельный lightweight-компонент для охранника.

### [SEC-3] 🟡 Разделение security vs executor нечёткое

`security` — это специализированный executor по коду. Меню почти одинаковое. Должен ли security видеть заявки на обслуживание? Если нет — нужна чётче ролевая изоляция.

## 6. РОЛЬ: МЕНЕДЖЕР (MANAGER)

### Хронология пользователя

```
Логин → ManagerDashboard (KPI) → Заявки → Назначить исполнителя
              ↓                        ↓
           Жители                 Здания → Подъезды → ...
              ↓
         Объявления → Создать
              ↓
         Собрания → Повестка → Голосование → Протокол
```

### [MGR-1] 🔴 Логика "Здания → Жители" разорвана

Менеджер в `/buildings` → видит дом → подъезд → квартиры. Чтобы посмотреть жителя — нужно выйти и перейти в `/residents`. **Большая навигационная проблема**.

Пользователь думает: "Я открыл дом → покажи жителей". Система говорит "иди в другой раздел".
**Рекомендация**: Buildings → Квартира → инлайн-список жителей (слайд-панель или вкладка). `/residents` остаётся для глобального поиска.

### [MGR-2] 🔴 Меню менеджера перегружено (21 пункт в 7 секциях)

На мобильном нужно скроллить сайдбар чтобы найти нужный пункт. Нарушает принцип "3 клика до любого экрана".

**Текущая структура**: Дашборд, Заявки, Наряды, Чат, Исполнители, Жители, Коллеги, Здания, Поиск авто, Гостевой доступ, Аренда, Объявления, Собрания, Тренинги, Маркетплейс (2), Финансы (6), Платежи, Отчёты, Блокнот, Настройки = **21 пункт**

**Рекомендация — группировка через collapsible-секции:**
```
[Главная] [Операции ▼] [Люди ▼] [Объект ▼] [Финансы ▼] [Настройки]
```
Каждая секция раскрывается при клике.

### [MGR-3] 🔴 Финансы без контекста квартиры/жителя

Менеджер в `/finance/charges` не видит привязки к конкретному жителю. Нет перехода "Квартира 54 → Долги по квартплате".
**Рекомендация**: из карточки квартиры/жителя — прямой переход к финансам этого объекта.

### [MGR-4] 🟡 Собрания — нет индикатора прогресса

Процесс создания собрания многошаговый (создать → вопросы → голосование → кворум → закрыть) но нет wizard-интерфейса. Менеджер должен помнить, где он в процессе.
**Рекомендация**: "Собрание #5" → статус-индикатор (Создано → Открыто → Голосование → Завершено). Следующий шаг подсвечен кнопкой CTA.

### [MGR-5] 🟡 Тренинги в основном меню — редкое действие

Тренинги нечастое действие, захламляет меню.
**Рекомендация**: убрать в Settings → HR или в раздел "Сотрудники".

### [MGR-6] 🟡 Аренда логически не в "Объектах"

`/rentals` находится рядом с "Поиск авто" и "Гостевой доступ". Логически ближе к Финансам или Жителям.
**Рекомендация**: переместить в секцию Финансы или создать секцию "Аренда и договоры".

## 7. РОЛЬ: АДМИНИСТРАТОР (ADMIN)

### [ADM-1] 🔴 Мониторинг системы в основном меню

`/monitoring` — системные метрики (health check, логи Workers). Это девопс-инструмент, его место — Settings → System, а не в боковой панели рядом с "Объявлениями".
**Рекомендация**: убрать из основного меню, переместить в Settings → "Системный мониторинг".

### [ADM-2] 🔴 Team + Buildings + Residents — три отдельных острова без связи

Нет "Карточки объекта" — единой страницы здания со всеми данными.
**Рекомендация**: concept "Building Card" — страница здания с вкладками:
```
[Жители] [Сотрудники] [Заявки] [Финансы] [Собрания]
```
Все данные в контексте одного объекта.

### [ADM-3] 🟡 AdminDashboard vs DirectorDashboard — разница неочевидна

Две роли с похожими меню и небольшими отличиями. Пользователи путаются в ролях.
**Рекомендация**: чётко документировать разделение: Admin = операционное управление, Director = стратегический overview. Это должно ощущаться в разных акцентах Dashboard.

## 11. ОБЩИЕ СИСТЕМНЫЕ ПРОБЛЕМЫ

### [SYS-1] 🔴 Отсутствует глобальный поиск

79 маршрутов, 13 ролей — и нет поиска по системе. Менеджер хочет найти жителя по фамилии — идёт в `/residents` и ищет там. Нет quick search.
**Рекомендация**: `Ctrl+K` (desktop) / кнопка поиска в Header (mobile) → модальный quick search по жителям, заявкам, исполнителям, зданиям.

### [SYS-2] 🔴 Нет явной 404-страницы

В роутинге нет `<Route path="*">`. При переходе на несуществующий путь — неизвестное поведение (белый экран или redirect).
**Рекомендация**: добавить 404-страницу с кнопкой "На главную".

### [SYS-3] 🟡 Чат — один раздел для всех ролей без разделения контекста

- Житель: "связь с управляющей компанией"
- Менеджер: "командный чат + личные переписки"
- Исполнитель: "получение задач + отчётность"

Один чат для всех сценариев = перегруженная UX без чёткого разделения каналов по типу.
**Рекомендация**: явные вкладки каналов: "Рабочие" / "Жильцы" / "Системные уведомления".

### [SYS-4] 🟡 4 параллельных механизма уведомлений без единого центра

Пользователь видит уведомления в: Push notification, Badge на BottomBar, Toast при открытом приложении, Bell icon в MobileHeader. Нет единого Notification Center — пользователь не знает, где искать "всё непрочитанное".
**Рекомендация**: `/notifications` как "входящие" — все уведомления с контекстом и action-кнопкой прямо там. Deep link из Push → конкретная заявка/объявление.

### [SYS-5] 🟡 Polling каждые 15 сек — масштабируемость

`fetchPendingReschedules()` каждые 15 сек для каждого жителя. При 1000+ жителях = 1000 запросов/15 сек = ~67 req/s только для reschedule. Cloudflare Workers имеют лимиты бесплатного плана.
**Файл**: `src/frontend/src/pages/ResidentDashboard.tsx`
**Рекомендация**: заменить на Server-Sent Events (SSE) или WebSocket, или увеличить интервал до 60 сек.

### [SYS-6] 🟡 Модальные окна vs full-page на мобильном — нет единого паттерна

**Правило**:
- Короткие формы (1–3 поля) → bottom sheet
- Длинные формы (создание заявки с фото, создание объявления) → full-page modal с кнопкой "Назад"
- Подтверждения (удалить, одобрить) → compact dialog по центру

### [SYS-7] 🔵 Нет Dark Mode (при наличии settingsStore)

`settingsStore.ts` упоминает dark mode. Пользователи PWA ожидают уважения системных предпочтений `prefers-color-scheme`.
**Рекомендация**: реализовать или убрать из настроек (не обещать функцию, которой нет).

### [SYS-8] 🔵 Нет индикатора offline-режима

Когда пользователь теряет сеть — нет визуального сигнала в UI (кроме SW fallback). Критично для управляющих/исполнителей в подвалах/технических помещениях.
**Рекомендация**: тонкий баннер "Вы офлайн — изменения сохранятся при восстановлении сети".

## 14. ПРОДУКТОВОЕ МНЕНИЕ

**Kamizo — сильный продукт с хорошей технической базой**, но навигационная архитектура растёт "снизу вверх" — каждая новая функция просто добавляется пунктом в меню. Это работает до ~12–15 пунктов, после — пользователь теряется.

### Ключевой тезис

**Пользователь работает с объектами** (здание, квартира, заявка, житель), **а не с разделами меню**. Навигация должна быть контекстно-связанной.

### Идеальный поток для менеджера (как должно быть)

```
Открыл "ЖК Камизо" →
  вижу подъезды →
    клик на подъезд →
      вижу квартиры →
        клик на квартиру 54 →
          [Жители] [Заявки] [Долги] [История гостей] [Договоры]
```

Сейчас эти данные разбросаны по 4–5 разным страницам. Это главная архитектурная задача следующего этапа.

### Сильные стороны

- **PWA** — технически отличная реализация (SW v2.1.0, offline, push, background sync)
- **Онбординг жителя** — логичная последовательность 4 шагов
- **Разделение Dashboard/Requests** у жителя — правильный UX-паттерн
- **Ролевая система** — 13 ролей с гибкой настройкой доступа
- **Multi-tenancy** — масштабируемая архитектура
- **i18n** — полный ru/uz

### Слабые стороны

- **Навигационные силосы** — данные не связаны между разделами
- **Мобильные модалки** — не адаптированы (bottom sheets)
- **Главный CTA жителя** — скрыт за BottomBar
- **Меню менеджера** — 21 пункт без группировки
