**Общий объём кода:** ~27 кБ (без node_modules).

### 1.2 Навигационная структура

Файл: `src/navigation/AppNavigator.tsx`

Используются два навигатора:

- **Stack Navigator** (`@react-navigation/native-stack`) — верхний уровень, переключает между Login и Main
- **Bottom Tab Navigator** (`@react-navigation/bottom-tabs`) — главный интерфейс с 3 вкладками

Структура:

```
Stack.Navigator (headerShown: false)
├── [не авторизован] → LoginScreen
└── [авторизован] → MainTabs (Bottom Tab Navigator)
    ├── Tab: "Home"       → HomeScreen (заглушка — инлайн-компонент, строка ~60)
    ├── Tab: "Colleagues" → ColleaguesScreen
    └── Tab: "Profile"    → ProfileScreen (заглушка — инлайн-компонент, строка ~66)
```

Логика переключения между стеками основана на `useAuthStore().user`: если `user !== null` — показывает MainTabs, иначе — LoginScreen. Нет промежуточного splash screen или проверки сохранённого токена при запуске.

### 1.3 Разделение по ролям

**Разделения по ролям НЕТ.** Приложение единое для всех пользователей. В типе `User` определены роли: `admin`, `manager`, `executor`, `resident`, `tenant`, `commercial_owner` — но нигде в коде навигации или экранов эти роли не проверяются. Все авторизованные пользователи видят одинаковые 3 таба.

### 1.4 Стартовый экран

При первом запуске (user === null) пользователь попадает на **LoginScreen**. После авторизации — на **MainTabs** (первая вкладка — Home, которая является заглушкой с текстом "Главная страница (в разработке)").

### 1.5 Экран логина

Файл: `src/screens/LoginScreen.tsx`

Поля формы:
- **Логин** (`login`) — текстовое поле, `autoCapitalize="none"`, `autoCorrect={false}`
- **Пароль** (`password`) — текстовое поле, `secureTextEntry`

Кнопка "Войти" вызывает `useAuthStore().login(login, password)`, который делает `POST /api/auth/login` с телом `{ login, password }`.

После входа: `user` устанавливается в store → навигация автоматически переключается на MainTabs.

**Проблема:** в бэкенде `POST /api/auth/login` ожидает поля `login` и `password` — это совпадает с мобильным приложением.

### 2.1 LoginScreen

**Файл:** `src/screens/LoginScreen.tsx` (3302 байт, ~95 строк кода + стили)
**Назначение:** Аутентификация пользователя
**Для какой роли:** Все роли (единый вход)

#### Данные
- **API вызов:** `POST /api/auth/login` (через `authAPI.login()` в `src/api/client.ts`)
- **Параметры запроса:** `{ login: string, password: string }`
- **Формат ответа:** `{ token: string, user: User }` (ожидается store'ом)
- **Кэширование:** Нет. Токен сохраняется в `global.authToken` (переменная JavaScript в памяти). При перезапуске приложения авторизация теряется.
- **Обработка ошибок:**
  - Пустые поля: `Alert.alert('Ошибка', 'Заполните все поля')` (клиентская валидация)
  - 401/ошибка сервера: `Alert.alert('Ошибка', 'Неверный логин или пароль')` (catch блок)
  - В store также устанавливается `error` из `error.response?.data?.message`

#### UI элементы
- **Лого:** Эмодзи 🏢 в оранжевом квадрате (80x80, borderRadius 20)
- **Заголовок:** "UK CRM"
- **Подзаголовок:** "Система управления жилым комплексом"
- **Поле "Логин"** — компонент Input с label
- **Поле "Пароль"** — компонент Input с secureTextEntry
- **Текст ошибки** — красный текст под полями (отображается если `error` из store не null)
- **Кнопка "Войти"** — компонент Button, показывает ActivityIndicator при `isLoading`
- **Футер:** "Версия 1.0.0 | © 2024 UK CRM"
- **Pull-to-refresh:** Нет
- **Пагинация:** Нет
- **Модальные окна:** Нет (используется Alert)
- **Loading state:** Кнопка "Войти" показывает спиннер
- **Empty state:** Н/П

#### Навигация
- После успешного входа: автоматический переход на MainTabs (через условие в AppNavigator)
- Нет экрана регистрации, восстановления пароля, выбора языка

#### Проблемы
1. Нет возможности зарегистрироваться — только вход
2. Нет "Забыли пароль?"
3. Нет выбора языка (ru/uz) — хотя в веб-версии двуязычность реализована

## 5. ПОТОКИ ДАННЫХ (DATA FLOWS)

### 5.1 Авторизация (единственный рабочий поток)

```
1. Пользователь вводит логин + пароль на LoginScreen
2. Нажимает "Войти"
3. useAuthStore.login() вызывает authAPI.login(login, password)
4. axios POST /api/auth/login → { login, password }
5. Бэкенд проверяет credentials → возвращает { token, user }
6. token сохраняется в global.authToken (глобальная JS-переменная)
7. user + token сохраняются в Zustand store (authStore)
8. AppNavigator реагирует на user !== null → рендерит MainTabs вместо LoginScreen
```

**Критические проблемы потока:**
- Токен хранится в `global.authToken` — теряется при перезапуске приложения
- Нет сохранения токена в SecureStore / AsyncStorage / MMKV
- Нет проверки истёкшего токена при запуске
- Нет refresh token механизма
- При 401 ответе interceptor только очищает `global.authToken`, но не перенаправляет на LoginScreen (это происходит косвенно — `user` остаётся в store)

### 5.2 Все остальные потоки
**НЕ РЕАЛИЗОВАНЫ.** Ниже перечислено, что должно быть:

- Регистрация жителя → создание аккаунта → привязка к квартире → первый вход — **НЕТ**
- Создание заявки жителем → уведомление УК → назначение → выполнение → закрытие → оценка — **НЕТ**
- Передача показаний счётчика → валидация → расчёт расхода → начисление — **НЕТ**
- Оплата → выставление счёта → оплата → обновление баланса — **НЕТ**
- Чат → отправка сообщения → доставка → прочтение — **НЕТ**
- Push-уведомление → создание УК → доставка на телефон жителя — **НЕТ**

## 7. STORES И STATE MANAGEMENT

### 7.1 Стейт-менеджер

**Zustand** (v5.0.9) — один файл store.

### 7.2 Список stores

| Store | Файл | Что хранит |
|---|---|---|
| `useAuthStore` | `src/stores/authStore.ts` | `user: User \| null`, `token: string \| null`, `isLoading: boolean`, `error: string \| null` |

**Это единственный store в приложении.**

### 7.3 Обновление стейта

`authStore.login()`:
1. Устанавливает `isLoading: true, error: null`
2. Вызывает `authAPI.login(username, password)`
3. При успехе: сохраняет `user`, `token`, `global.authToken`
4. При ошибке: сохраняет `error` из `error.response?.data?.message`

`authStore.logout()`:
1. Очищает `global.authToken = undefined`
2. Устанавливает `user: null, token: null`

### 7.4 Персистентное хранилище

**НЕТ.** Нет использования AsyncStorage, SecureStore, MMKV или любого другого персистентного хранилища. Токен аутентификации хранится в `global.authToken` (переменная JavaScript runtime) — при закрытии приложения токен теряется, пользователю нужно заново логиниться каждый раз.

### 7.5 Хранение токена

Токен хранится в двух местах:
1. `global.authToken` — глобальная переменная, используется axios interceptor'ом для добавления `Authorization: Bearer` header
2. `authStore.token` — состояние Zustand store, нигде не читается кроме самого store

Это **серьёзная проблема безопасности и UX**: токен должен храниться в `expo-secure-store` (SecureStore) и восстанавливаться при запуске.

# 1 из 10
